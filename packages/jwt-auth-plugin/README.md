# @andrey-aka-skif/jwt-auth-plugin

Плагин JWT-аутентификации для SPA на Vue 3 + Vue Router. Берёт на себя хранение
токенов, восстановление и поддержание сессии, защиту маршрутов и подстановку
токена в запросы — поверх вашего экземпляра axios.

Peer-зависимости: `vue` (^3), `vue-router` (^4), `axios`.

## Возможности

- Хранение пары токенов в `localStorage` (с пространством имён).
- Восстановление сессии при загрузке и автоматический рефреш — по таймеру и на 401.
- Сериализация рефреша через Web Locks: один рефреш на все вкладки + межвкладочная
  синхронизация сессии.
- Гарды маршрутов: защита по `meta.auth`, редиректы, возврат на исходную страницу
  после логина.
- Единая классификация ошибок (`getErrorKind`) для UI и внутренней логики.

## Установка

```bash
npm install @andrey-aka-skif/jwt-auth-plugin
```

```js
import { createApp } from 'vue'
import auth from '@andrey-aka-skif/jwt-auth-plugin'
import router from './router'
import axiosInstance from './api/instance'

createApp(App)
  .use(router)
  .use(auth, {
    router,
    axiosInstance,
    config: {
      api: { baseURL: import.meta.env.VITE_API_BASE_URL },
      token: { access: { subKey: 'sub' } },
    },
  })
  .mount('#app')
```

## Как это работает

**Единый axios-инстанс.** Плагин навешивает на переданный `axiosInstance`
интерсепторы: request подставляет `Bearer <access>` в заголовок
(`token.access.requestKey`), а response на ошибку из `session.logoutStatuses`
(по умолчанию 401) запускает рефреш и **один раз** повторяет исходный запрос. Тот
же инстанс используете и вы (и сгенерированный SDK) — поэтому все запросы
аутентифицируются единообразно. Auth-эндпоинты (`login` / `refresh` / `logout` /
`userinfo`) плагин обслуживает сам поверх того же инстанса.

**Жизненный цикл сессии.** При установке (`plugin.autoStart`) плагин восстанавливает
сессию: читает access-токен из `localStorage` и, если сменился пользователь (claim
`token.access.subKey`), берёт данные пользователя из источника `session.userSource`
(по умолчанию — из claims самого токена, без сетевого запроса; см.
[Источник данных пользователя](#источник-данных-пользователя-sessionusersource)).
Пока восстановление не завершено, `isReady` = `false`, а гард маршрутов ждёт его
перед первой навигацией — это окно удобно показать пользователю как состояние
загрузки.

**Хранение и вкладки.** Пара токенов лежит в `localStorage` (под
`storage.namespace`). Обновление токена в одной вкладке подхватывается другими через
событие `storage`.

### Рефреш

- **Проактивно (по таймеру).** При `plugin.autoRefresh` шедулер раз в
  `token.refresh.checkIntervalMinutes` (с джиттером `checkJitterPercent`) проверяет
  access-токен и обновляет пару, если до истечения осталось меньше
  `checkIntervalThresholdMinutes`.
- **Реактивно (на 401).** Если запрос вернул статус из `session.logoutStatuses`,
  интерсептор делает рефреш и **один раз** повторяет исходный запрос (флаг `_retry`
  исключает циклы).
- **Один рефреш на все вкладки.** Рефреш идёт под Web Lock
  (`token.refresh.lockKey`, тайм-аут `lockTimeout`), поэтому параллельные запросы и
  вкладки не запускают рефреш одновременно.
- **Гонка вкладок.** Если рефреш вернул auth-ошибку (например, refresh-токен уже
  обновила другая вкладка), плагин перечитывает хранилище до `raceWaitMaxAttempts`
  раз с интервалом `raceWaitIntervalMs` и использует уже обновлённый соседом токен
  вместо разлогина.
- **Сетевой сбой.** При `session.keepSessionOnNetworkError` сетевая ошибка во время
  рефреша не рвёт сессию — попытка повторится на следующем тике шедулера.

### Технические подробности

> Дальше — рабочая модель того, как это ведёт себя в браузерах: собрана из
> наблюдений и устройства реализаций, а не строго гарантированный спецификацией
> факт. Но картина весьма вероятная и объясняет, почему синхронизация устроена
> именно так.

- **`storage`-событие не приходит во вкладку-источник.** По спецификации оно
  доставляется только в _другие_ вкладки того же origin. Поэтому вкладка, которая
  сама обновила токен, обрабатывает это напрямую в потоке рефреша, а не ждёт
  собственного события.
- **Доставка `storage` — асинхронная и межпроцессная (IPC).** Современные браузеры
  изолируют вкладки по процессам; `localStorage` общий для origin, но оповещение
  других вкладок идёт через IPC браузера. Отсюда нет жёстких гарантий тайминга и
  порядка относительно самой записи: между записью в одной вкладке и событием в
  другой возможна задержка, а серия быстрых записей может «схлопнуться» до
  последнего значения.
- **Поэтому событие — не единственный источник истины.** Если рефреш упал с
  auth-ошибкой, вкладка не доверяет только событию (которое могло ещё не прийти), а
  несколько раз перечитывает `localStorage` **напрямую** (синхронное чтение,
  `raceWaitMaxAttempts` × `raceWaitIntervalMs`) — чтобы подхватить токен, который
  сосед уже записал, но событие про это ещё «в пути».
- **Сериализация рефреша тоже межпроцессная.** `navigator.locks` координирует
  вкладки через тот же IPC-механизм; блокировка снимается, когда контекст-держатель
  исчезает (закрытие/креш вкладки), а `lockTimeout` страхует от бесконечного
  ожидания, если держатель «завис».
- **Окно гонки сохраняется.** На весь цикл read-modify-write нет общей транзакции:
  две вкладки могут прочитать один и тот же протухающий токен. Web Lock сужает это
  окно (рефреш делает кто-то один), а перечитывание выше гасит остаток.

## Конфигурация

Обязательны два поля:

- `api.baseURL` — базовый URL бэкенда;
- `token.access.subKey` — claim в access-токене, идентифицирующий пользователя
  (по нему плагин замечает смену пользователя, в т. ч. между вкладками).

Остальное имеет значения по умолчанию — полный список в
[`src/core/defaultConfig.js`](src/core/defaultConfig.js). Наиболее полезное:

| Параметр                                                 | По умолчанию                     | Назначение                                                                                 |
| -------------------------------------------------------- | -------------------------------- | ------------------------------------------------------------------------------------------ |
| `endpoints.{login,refresh,logout,userinfo}`              | `/api/auth/*`                    | Пути auth-эндпоинтов (относительно `api.baseURL`)                                          |
| `storage.namespace`                                      | имя пакета                       | Префикс ключей localStorage; `''` / `null` — без префикса                                  |
| `storage.accessTokenKey` / `refreshTokenKey`             | `access-token` / `refresh-token` | Ключи localStorage для токенов                                                             |
| `token.access.requestKey`                                | `Authorization`                  | Заголовок, в который кладётся `Bearer <access>`                                            |
| `token.access.responseKey` / `token.refresh.responseKey` | `access_token` / `refresh_token` | Ключи токенов в теле ответа                                                                |
| `token.refresh.requestKey`                               | `refresh_token`                  | Имя поля/заголовка с refresh-токеном (`'refresh_token'` \| `'X-Refresh-Token'`)            |
| `token.refresh.requestMethod`                            | `body`                           | Как передавать refresh-токен: `'body'` \| `'header'`                                       |
| `session.userSource`                                     | `token`                          | Откуда брать данные пользователя: `'claims'` (claims токена) \| `'endpoint'` (`/userinfo`) |
| `session.logoutStatuses`                                 | `[401]`                          | Статусы ответа, означающие невалидную сессию → разлогин                                    |
| `session.keepSessionOnNetworkError`                      | `true`                           | Не рвать сессию при сетевой ошибке рефреша (повторить позже)                               |
| `redirect.onNotAuthenticated`                            | `{ name: 'login' }`              | Куда редиректить гостя с защищённого маршрута                                              |
| `redirect.backToPreviousOnAuthenticated.enabled`         | `true`                           | Возвращать после логина на исходный маршрут (через query)                                  |
| `plugin.autoStart` / `plugin.autoRefresh`                | `true` / `true`                  | Старт сессии при установке / фоновый рефреш по таймеру                                     |

## Источник данных пользователя (`session.userSource`)

Откуда `user` берёт данные при восстановлении сессии:

- **`'claims'`** (по умолчанию) — из декодированных claims access-токена, **без
  сетевого запроса**. `user.value` — это **весь payload токена как есть**: и
  прикладные claims, и служебные (`exp`, `iss`, `aud`, `iat`, `jti`, …), под теми
  именами, что использует ваш бэкенд (в т. ч. «некрасивые» ClaimTypes-URI вида
  `http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress`). Плагин
  ничего не фильтрует и не переименовывает — приведение к удобной для UI форме
  (короткие имена, отбор полей) остаётся на стороне приложения.
- **`'endpoint'`** — профиль запрашивается с сервера (`endpoints.userinfo`), `user.value`
  = тело ответа как есть. Прежнее (до этой опции) поведение плагина.

Что учесть при выборе `'claims'`:

- **Данные ограничены содержимым токена.** Если нужных полей в токене нет —
  используйте `'endpoint'` или запрашивайте профиль отдельным запросом из
  приложения.
- **Токен тяжелеет и шлётся с каждым запросом.** Класть в него много профильных
  данных ради экономии на `/userinfo` — плата размером заголовков на всех запросах.
- **Claims в JWT не шифруются** (только base64) — что не должно быть видно клиенту,
  сервер не должен класть в токен; для приватных данных используйте `'endpoint'`.
- **Свежесть.** `user` заполняется при восстановлении сессии и при смене
  пользователя (`subKey`); фоновый рефреш обновляет токены, но `user` не
  переписывает — ровно как `'endpoint'`-режим не перезапрашивает `/userinfo` на
  каждом рефреше. Это намеренная симметрия, а не потеря смысла: профильные claims
  (email, роль и т. п.) для того же пользователя между рефрешами не меняются, а
  технические (`exp`, `jti`, …) для UI бесполезны. Если профильные данные могут
  меняться в течение сессии, обновляйте их в приложении (перезапрос профиля /
  проверка на сервере), не полагаясь на кэш claims.

## Публичный API

`useAuth()` возвращает объект сессии:

```js
import { useAuth } from '@andrey-aka-skif/jwt-auth-plugin'

const {
  login,
  logout,
  refresh,
  user,
  isReady,
  isAuthenticated,
  lastError,
  getErrorKind,
} = useAuth()
```

| Поле                  | Назначение                                                                                                                                                                                |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `login(credentials)`  | Логин; `credentials` передаются в тело запроса на `endpoints.login` как есть — их форма определяется вашим бэкендом (например, `{ email, password }`). Пробрасывает исходную ошибку axios |
| `logout()`            | Выход (запрос на сервер + очистка сессии)                                                                                                                                                 |
| `refresh()`           | Ручной рефреш пары токенов                                                                                                                                                                |
| `user`                | `readonly`-ref с данными пользователя (или `null`)                                                                                                                                        |
| `isAuthenticated`     | `readonly`-ref: есть ли активная сессия                                                                                                                                                   |
| `isReady`             | `readonly`-ref: завершена ли инициализация сессии                                                                                                                                         |
| `lastError`           | `readonly`-ref: последняя ошибка рефреша                                                                                                                                                  |
| `getErrorKind(error)` | Классифицирует ошибку: `'auth' \| 'network' \| 'unknown'`                                                                                                                                 |

## Защита маршрутов

Гард читает `meta` маршрутов:

- `meta.auth: true` — маршрут требует аутентификации; гостя редиректит на
  `config.redirect.onNotAuthenticated` (по умолчанию `{ name: 'login' }`);
- `meta.redirectOnAuthenticated` — куда отправить уже аутентифицированного
  (например, со страницы логина);
- при `config.redirect.backToPreviousOnAuthenticated.enabled` исходный путь
  сохраняется в query (`redirect` по умолчанию) и после логина пользователь
  возвращается туда же (только безопасные внутренние пути).

## Обработка ошибки входа в UI

`login` пробрасывает исходную ошибку axios (не оборачивая её). Чтобы единообразно
её истолковать, используйте `getErrorKind`:

```js
const { login, getErrorKind } = useAuth()

try {
  await login({ email, password })
} catch (e) {
  switch (getErrorKind(e)) {
    case 'auth':
      showError('Неверный e-mail или пароль')
      break
    case 'network':
      showError('Сервер недоступен')
      break
    default:
      showError('Не удалось войти')
  }
}
```

`getErrorKind` — единая трактовка ошибок: `'auth'` (статус из
`session.logoutStatuses`) → переаутентификация, `'network'` (ответа нет) → при
`session.keepSessionOnNetworkError` сессия сохраняется и рефреш повторяется позже,
иначе — разлогин.

## Работа со сгенерированным SDK (@hey-api и т. п.)

Свои запросы (не auth) приложение делает через собственный SDK. Чтобы на них
действовали те же auth-интерсепторы (токен и рефреш на 401), достаточно, чтобы SDK
ходил через **тот же** `axiosInstance`, что передан плагину:

```js
import axios from 'axios'
import { client } from './api/generated/client.gen'

const axiosInstance = axios.create({ baseURL })

// SDK конфигурируем на тот же инстанс
client.setConfig({ axios: axiosInstance })

app.use(auth, { router, axiosInstance, config })
```

> Рабочий пример целиком — в `playgrounds/spa` (`shared/api/http/httpTransport.js`,
> `shared/api/setup/setupHeyApiClient.js`).
